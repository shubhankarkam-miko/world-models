# Inference-Time Filtering Mechanisms

This document details methods for implementing real-time, run-time safety filters that act as immediate vetoes on candidate actions generated by a VLA policy. These mechanisms are characterized by their low latency requirements and ability to check outputs instantly, making them ideal for fast, responsive systems.

## 🧠 Key Principles
The core principle is **Action Vetoing**: The filter receives the predicted action $\mathbf{a}_{candidate}$ (and associated state/context) and returns `SAFE` or `UNSAFE`. If unsafe, it might return a suggested corrective action $\mathbf{a}_{correction}$.

## 🛡️ Approaches and Implementations
### 1. Safety Q-Filter (VLESA Approach)
*   **Goal:** To classify whether an observed visual/action sequence is safe based on pre-trained models of hazard datasets.
*   **Mechanism:** Training a dedicated Q-filter model (analogous to a classifier or policy head trained specifically for safety). It learns the boundary between acceptable and prohibited physical actions.
*   **Best Used When:** The set of unsafe actions can be relatively well-defined and categorized into distinct types.
*   **Source Reference:** [VLA-Safety Agent for Human Activity Monitoring (VLESA)](https://arxiv.org/abs/2606.03954).

### 2. Attention-Guided Safety Filter (The Perceptual Guardrail)
*   **Goal:** To extract safety signals directly from the internal, uninterpretable features of the main VLA model itself, making the filter highly integrated and fast.
*   **Mechanism:** Instead of adding a massive external network, this method uses attention weights or latent vectors within the existing policy to calculate physical constraint violations (e.g., predicting if an action will cause joint hyper-extension). This is extremely efficient as it avoids multiple forward passes through large models.
*   **Best Used When:** High performance and minimal computational overhead are paramount. The safety signal must be derived *perceptually* from the current input features.
*   **Source Reference:** [Attention-Guided Safety Filter for VLA](https://arxiv.org/abs/2606.09749).

## ⚙️ Development Guidance for Agents
When implementing a filter, consider the following trade-offs:

| Criterion | Inference-Time Filtering | Implication |
| :--- | :--- | :--- |
| **Latency** | Very Low (Must be real-time) | Requires specialized hardware or highly optimized models. |
| **Scope of Failure Detection** | Limited to the immediate action space. | Cannot predict cascading failures far into the future. |
| **Implementation Difficulty** | Medium | Requires hooking into the model's forward pass structure. |

***Always test filters against adversarial examples and boundary condition data sets (like those used in VLA-RISK).***
